Schema Design

Normalization: 1NF → 3NF

Eliminating redundancy, partial & transitive dependencies

1NF

First Normal Form — Atomic Values

Problem: phone_no is multivalued — one user can have multiple phones. Fix: Decomposed into user_phones(user_id, phone_no) with composite primary key. Every cell now holds exactly one atomic value. ✅

2NF

Second Normal Form — No Partial Dependencies

Checked all composite-key tables: joins(user_id, group_id, join_date) — join_date depends on the FULL composite key, not just one part. follows and post_hashtags have no non-key attributes. All tables satisfy 2NF. ✅

3NF

Third Normal Form — No Transitive Dependencies

User details (bio, dob) stored ONLY in users — never repeated in posts or comments. No non-key attribute determines another non-key attribute in any table. Schema satisfies 3NF and BCNF for most relations. ✅

🏆

Result: Clean, Non-Redundant, Consistent Schema

No data anomalies. No update, insertion, or deletion anomalies. Referential integrity enforced via ON DELETE CASCADE on all foreign keys across all 11 tables.

Additional DBMS Concepts
  • Indexes on username, post_id, sender+receiver for fast lookups
  • ON DELETE CASCADE — referential integrity across all FKs
  • CHECK constraint — follower_id ≠ followee_id (no self-follow)
  • Transactions — user insert + phone insert are atomic
SQL Engineering

Advanced SQL Queries

5 mandatory query categories — all executed live from the web dashboard

JOIN

INNER + LEFT JOIN

Posts with author username and primary phone number — LEFT JOIN includes users with no phone (NULL)

SELF JOIN

Recursive Self Join

Mutual followers — follows table joins itself to detect bidirectional edges in the social graph

SUBQUERY

Nested Subquery

Users who commented but never posted — uses IN and NOT IN correlated subqueries

SELECT u1.username AS user_a, u2.username AS user_b FROM follows f1 INNER JOIN follows f2 ON f1.follower_id = f2.followee_id AND f1.followee_id = f2.follower_id WHERE f1.follower_id < f1.followee_id;
AGG

Aggregate Functions

COUNT of likes, comments & hashtags per post using scalar subqueries inside SELECT

HAVING

GROUP BY + HAVING

Groups with 3+ members — HAVING filters after aggregation (unlike WHERE which filters before)
